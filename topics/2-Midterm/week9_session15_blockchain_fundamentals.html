What If...

What if you could verify transactions WITHOUT a middleman? Without trusting any single authority?

06

What Is Blockchain?

Blockchain

A distributed, immutable digital ledger that records transactions across many computers. No single computer controls it. Once data is recorded, it cannot be altered without changing every subsequent block AND getting majority agreement from the network.

Centralized vs Decentralized

Centralized (Traditional)

  • ONE authority controls the ledger
  • Bank holds THE record of your money
  • Single point of failure
  • Trust the institution
  • Can be censored or frozen
  • Example: BDO, GCash, government

Decentralized (Blockchain)

  • EVERYONE holds a copy of the ledger
  • All participants verify transactions
  • No single point of failure
  • Trust the math/protocol
  • Cannot be censored or frozen
  • Example: Bitcoin, Ethereum
09

The Notebook Analogy

Imagine a class notebook tracking who owes whom money:

Centralized

Teacher holds THE notebook. Students trust teacher. What if teacher erases a record? Lost. What if notebook is stolen? Disaster.

Blockchain

EVERY student has an exact copy. Any change must be verified by majority. If one copy is altered, 39 others disagree. Impossible to cheat.

Hashing Explained

Hash = Digital Fingerprint

A hash function takes ANY input (text, file, data) and produces a fixed-length unique output. Change ONE character in the input? Completely different hash output. Like a fingerprint: unique to that exact data.

"Hello" → 2cf24dba5fb0... | "Hello!" → 33b79f... (totally different!)

13

The Chain: Why Tampering Fails

Each block contains the hash of the PREVIOUS block. Change one block? Its hash changes. Now the NEXT block's "previous hash" doesn't match. That breaks the entire chain from that point forward.

Block 1 (hash: ABC) → Block 2 (prev: ABC, hash: DEF) → Block 3 (prev: DEF, hash: GHI)

Change Block 2? Its hash changes to XYZ. Now Block 3 expects "DEF" but gets "XYZ." INVALID. The network rejects it.

The Agreement Problem

With no central authority, how does everyone agree on the "truth"? If 10,000 computers have the ledger, who decides which transactions are valid?

Answer: CONSENSUS MECHANISMS. Mathematical rules that force agreement without trust.

18

PoW vs PoS

Proof of Work (PoW)

  • Miners solve complex math puzzles
  • First to solve adds the block
  • Requires massive computing power
  • Very secure but energy-hungry
  • Used by: Bitcoin

Proof of Stake (PoS)

  • Validators "stake" their coins as collateral
  • Selected based on stake size + randomness
  • No mining puzzles needed
  • 99% less energy than PoW
  • Used by: Ethereum (since 2022)
19

PoW: Mining Explained

Miners race to find a special number (nonce) that, when combined with block data and hashed, produces a result with a specific pattern (e.g., starts with four zeros).

Analogy: Like a lottery where you can't buy tickets faster, only try more combinations. The first miner to find the winning number gets to add the block AND earns a reward (currently 3.125 BTC ≈ $200,000+).

Bitcoin ≠ Blockchain

Bitcoin is an APPLICATION of blockchain. Blockchain is the underlying TECHNOLOGY.

Like email is an application of the internet. The internet can do much more than email. Blockchain can do much more than Bitcoin.

Smart Contracts (Preview)

Smart Contract

Self-executing code stored on blockchain that automatically enforces agreement terms when conditions are met. No lawyers, no intermediaries. "If X happens, then automatically do Y."

Example: Freelancer contract. Code says: "When client confirms delivery, automatically release payment." No disputes, no delays.

29

Blockchain Limitations

  • Speed: Bitcoin processes ~7 transactions/sec (Visa: 65,000/sec)
  • Energy: PoW mining consumes massive electricity
  • Scalability: more users = slower, more expensive
  • Irreversibility: no undo button (mistakes are permanent)
  • Complexity: hard for average users to understand/use
  • Regulation: unclear laws in many countries
30

When Blockchain Makes Sense

Good Use Cases ✓

  • Multiple parties need shared truth
  • Trust between parties is low
  • Intermediary is expensive/slow
  • Audit trail is critical
  • Data shouldn't be altered

Bad Use Cases ✗

  • Single organization controls data
  • Trust already exists
  • Speed is critical (high TPS needed)
  • Data needs to be edited/deleted
  • Simple database would work fine
