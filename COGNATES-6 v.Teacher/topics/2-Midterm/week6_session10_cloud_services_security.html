Computing

12

What Is Serverless?

Serverless ≠ No Servers

Servers still exist, but YOU don't see or manage them. You write code, upload it, and the cloud runs it automatically when triggered. Pay ONLY when code executes. Zero cost when idle.

13

Serverless vs Traditional

Traditional Server

  • Server runs 24/7
  • Pay even when idle
  • You manage OS, patches
  • You handle scaling
  • Predictable monthly cost

Serverless

  • Runs only when triggered
  • Pay per execution (milliseconds)
  • Zero maintenance
  • Auto-scales to any load
  • Unpredictable cost (usage-based)

The Shared Responsibility Model

16

Who Secures What?

Cloud security is SHARED between the provider and the customer.

Provider secures the infrastructure. YOU secure your data and applications.

17

Shared Responsibility

Provider SecuresCustomer Secures
Physical data centersYour data (encryption)
Hardware (servers, disks)Access control (who can log in)
Network infrastructureApplication security (code vulnerabilities)
Hypervisor/virtualizationOS patches (for IaaS)
Global backboneFirewall rules and configurations
18

Security by Service Model

ModelProvider ResponsibilityYour Responsibility
IaaSHardware + network onlyOS, apps, data, identity, everything else
PaaSHardware + network + OS + runtimeApplication code + data + identity
SaaSAlmost everythingData + user access + settings

More SaaS = less responsibility for you, but also less control.

19

Common Cloud Security Mistakes

  • Leaving storage buckets PUBLIC (data exposed to internet)
  • Weak passwords / no MFA on admin accounts
  • Not encrypting sensitive data at rest
  • Overly permissive access (everyone is admin)
  • Ignoring security alerts and logs
  • No backup strategy (provider won't recover your deleted data)

The 6 R's: How organizations move to cloud

23

The 6 R's of Migration

StrategyWhat It MeansExample
Rehost (Lift & Shift)Move as-is to cloudCopy VM to AWS EC2

RepurchaseSwitch to SaaS alternativeReplace custom CRM with Salesforce

RetainKeep on-premises (for now)Legacy systems too complex to move
24

Most Common: Lift & Shift

Fastest way to migrate. Take your existing system and move it to cloud VMs with minimal changes. Not optimal, but gets you to cloud quickly.

Analogy: Moving house. You pack everything in boxes and move to a new house. You don't renovate first. Just get there. Optimize later.

25

Best Long-Term: Refactor

Rebuild your application to be cloud-NATIVE. Use containers, microservices, serverless. Maximum cloud benefits, but expensive and time-consuming.

Analogy: Instead of moving to a new house, you DESIGN and BUILD a new house specifically for your needs. Better result, but takes way longer.

Key Takeaways
